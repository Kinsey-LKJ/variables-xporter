# Principles
整理变量是保证 Variables Xporter 能够成功导出用于开发环境代码的关键。请现在 Figma 中按照以下原则整理变量。

## 命名原则

为了让变量名具有更好的可读性和可维护性，应当遵循以下原则：

- 使用英文，因为你的变量最终将导出为 Tailwind CSS 配置文件或 CSS 变量文件。
- 语义化命名，能够清晰地表达变量的用途。
- 简洁明了，避免冗长和复杂的命名。


{/* <Callout type="info" emoji="ℹ️">
  在使用 Varibales Xporter 时，你必须使用 camelCase 或者 kebab-case 命名格式将多个单词分隔，例如 `borderRadius`，或者 `border-radius`。
</Callout> */}

### 对于 Tailwind CSS 项目
如果你导出的变量用于 Tailwind CSS 项目，变量的顶层名称应当是 Tailwind CSS 的主题配置键，例如：

- `colors`
- `spacing`
- `fontSize`
- `fontWeight`
- `borderRadius`
- `opacity`

你可以在 [Tailwind CSS 主题配置参考](https://tailwindcss.com/docs/theme#configuration-reference) 中查看完整的配置键列表。


## 分隔原则
当某个组件名称或样式名称包含多个单词时，请使用以下格式将它们分隔：
- camelCase -> twoWords
- constantCase -> TWO_WORDS
- kebabCase -> two-words
- pascalCase -> TwoWords
- pascalSnakeCase -> Two_Words
- snakeCase -> two_words
- trainCase -> Two-Words
- capitalCase -> Two Words
- sentenceCase -> Two words

<Callout type="info" emoji="ℹ️">
  在导出变量时，Varibales Xporter 会自动将变量名统一转换为符合 Tailwind 配置文件以及 CSS 变量命名规则或习惯的格式。
</Callout>


## 分组原则
请使用 `/` 将变量进行合理的分组，这样可以更好的组织你的变量，使其更加清晰易读，同时也方便其他设计师和开发人员查找和使用。
就像整理文件一样，你不会把所有文件都丢在电脑的根目录里，而是将他们放在不同的文件夹，例如：

<FileTree>
  <FileTree.Folder name="colors" defaultOpen>
    <FileTree.File name="primary" />
    <FileTree.File name="secondary" />
    <FileTree.File name="tertiary" />
    <FileTree.File name="..." />
  </FileTree.Folder>
</FileTree>

这分别对应以下 Figma 中的变量：

- `colors/primary`
- `colors/secondary`
- `colors/tertiary`

### 使用默认值组织变量层级 [#use-default-value]
在有些时候，你可能需要使用默认值，例如以下变量：

- `colors/primary`
- `colors/primary/foreground`

仅这样命名有时是不够的，我们依然用文件管理的方式来思考，例如：

<FileTree>
  <FileTree.Folder name="colors" defaultOpen>
    <FileTree.File name="primary" />
    <FileTree.Folder name="primary" defaultOpen>
      <FileTree.File name="foreground" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

这会出现一个问题，你无法在一个目录同时管理 `colors/primary` 和 `colors/primary/foreground` 两个变量，尽管你认为他们应该出现在一起。
在 Figma 中也会出现同样的情况，在这里无法看到 `colors/primary` 这个变量:

想想当我们的变量变得越来越多时，要想找到 `colors/primary` 这个变量，会变得更麻烦，**因为我们无法依靠左边栏快速定位，而是要在 colors 这个目录下寻找**，
但仅仅只有管理上的问题？当我们在 Tailwind CSS 的配置文件中还原这个场景时，会发现问题更大，我们无法拥有两个相同的键：

```ts twoslash
// @errors: 1117
export default {
  theme: {
    extend: {
      colors: {
        primary: 'var(--colors-primary)',
        primary: {
          foreground: 'var(--colors-primary-foreground)',
        },
      },
    },
  },
  plugins: [],
};

```

使用默认值 `DEFAULT` 可以完美的解决这个问题：
```ts twoslash
// @errors: 1117
export default {
  theme: {
    extend: {
      colors: {
        primary: {
          DEFAULT: 'var(--colors-primary)',
          foreground: 'var(--colors-primary-foreground)',
        },
      },
    },
  },
  plugins: [],
};
```
并且在使用 `colors.primary.DEFAULT` 这个工具类时，Tailwind CSS 会自动省略 `DEFAULT`，你只需要像这样：

```html
<div class="text-primary">
  <p>Hello</p>
</div>
```

在 Figma 中就可以借住 `DEFAULT` 这个默认值标记，保证某些变量处于同一层级：

- `colors/primary/DEFAULT`
- `colors/primary/foreground`

他们便会出现在同一个目录下：
