# 基本原则

在使用 Variables Xporter 时，变量的组织方式会直接影响导出代码的质量。让我们来探讨如何在 Figma 中有效地组织变量，以确保导出的代码能够无缝集成到您的开发环境中。

## 命名原则

为了使变量更易于理解和维护，我们建议遵循以下命名原则：

- 使用英文命名：这可以确保与开发环境的兼容性，并在导出到 Tailwind CSS 配置或 CSS Variables 时保持一致性
- 选择语义化命名：变量名应清晰表达其用途，使团队成员能够快速理解
- 保持简洁：避免过长或复杂的命名，以维护代码的可读性和可维护性

{/* <Callout type="info" emoji="ℹ️">

</Callout> */}

## 命名约定

在设计系统中，我们经常需要使用多个单词来描述一个变量。Variables Xporter 会根据目标格式自动转换命名格式：

- Tailwind CSS 配置文件：使用 camelCase 格式（例如：fontSize, fontWeight）
- CSS 变量：使用 kebab-case 格式（例如：font-size, font-weight）

以下是一些常见的命名格式供参考：
- camelCase -> twoWords
- constantCase -> TWO_WORDS
- kebabCase -> two-words
- pascalCase -> TwoWords
- pascalSnakeCase -> Two_Words
- snakeCase -> two_words
- trainCase -> Two-Words
- capitalCase -> Two Words
- sentenceCase -> Two words

<Callout type="info" emoji="ℹ️">
  在导出变量时，无论您使用上述哪种格式，Variables Xporter 都会自动将输出转换为符合 `tailwind.config.js` 文件和 CSS 变量的约定或要求：
  - `tailwind.config.js` 文件：使用 camelCase 格式（例如：fontSize, fontWeight）
  - CSS 变量：使用 kebab-case 格式（例如：font-size, font-weight）
</Callout>

## 分组原则

合理的变量分组可以显著提高工作流程的效率。我们建议使用 `/` 符号来组织变量层级，类似于管理文件系统：

想象一下，如果所有文件都堆在计算机的根目录中，寻找文件会有多困难。同样，我们需要为变量创建清晰的层级结构：

<FileTree>
  <FileTree.Folder name="colors" defaultOpen>
    <FileTree.File name="primary" />
    <FileTree.File name="secondary" />
    <FileTree.File name="tertiary" />
    <FileTree.File name="..." />
  </FileTree.Folder>
</FileTree>

这些对应于 Figma 中的以下变量：

- `colors/primary`
- `colors/secondary`
- `colors/tertiary`

### 使用 `DEFAULT` 关键字组织变量层级 [#using-default-keyword]

`DEFAULT` 关键字有双重作用：它设置默认值，更重要的是，它统一了同一级别的变量，使变量管理更加清晰和系统化。让我们看一个具体的例子：

- `colors/primary`
- `colors/primary/foreground`

这是一个常见的变量集，用于主色和主背景上的文本颜色，但仅靠这种命名还不够。让我们从文件管理的角度来思考：

<FileTree>
  <FileTree.Folder name="colors" defaultOpen>
    <FileTree.File name="primary" />
    <FileTree.Folder name="primary" defaultOpen>
      <FileTree.File name="foreground" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

这里我们遇到了一个有趣的挑战：虽然逻辑上 `colors/primary` 和 `colors/primary/foreground` 应该在同一个目录中，但实际上它们分散在不同的位置。让我们看看在 Figma 中是如何显示的：

随着变量数量的增加，找到 `colors/primary` 变得更加繁琐，**因为我们无法依赖侧边栏快速导航，而必须在 colors 目录中搜索**。
这不仅影响了我们的工作效率，更重要的是，当我们尝试将这种结构转换为 Tailwind CSS V3 配置时，我们遇到了一个技术限制：**JavaScript 对象不能同时将同一个键用作值和对象**。

```ts twoslash
// @errors: 1117
export default {
  theme: {
    extend: {
      colors: {
        primary: 'var(--colors-primary)', 
        primary: {
          foreground: 'var(--colors-primary-foreground)',
        },
      },
    },
  },
  plugins: [],
};
```

不过不用担心，Tailwind CSS V3 提供了一个优雅的解决方案——使用 `DEFAULT` 关键字：

```ts twoslash
// @errors: 1117
export default {
  theme: {
    extend: {
      colors: {
        primary: {
          DEFAULT: 'var(--colors-primary)',
          foreground: 'var(--colors-primary-foreground)',
        },
      },
    },
  },
  plugins: [],
};
```

最棒的是，在实际开发中，您不需要手动写出 `DEFAULT` 关键字。Tailwind CSS 会自动处理这个细节，保持代码简洁：

```html
<div class="text-primary">
  <p>Hello, world!</p>
</div>
```

通过这种技术，我们可以优雅地在 Figma 中组织变量。只需使用 `DEFAULT` 作为默认值的标记：

- `colors/primary/DEFAULT` - 主色的默认值
- `colors/primary/foreground` - 主背景上的文本颜色

它们将出现在同一个目录中：

在最新的 Tailwind CSS V4 中，配置文件已从 JavaScript 文件迁移到 CSS 文件，因此不再需要 `DEFAULT` 关键字。然而，我们仍然建议使用 `DEFAULT` 关键字来组织变量层级，以确保 Figma 中变量的层级结构清晰。当您选择 Tailwind CSS V4 或 shadcn/ui (Tailwind CSS V4) 作为输出格式时，Variables Xporter 会自动从导出结果中移除多余的 `DEFAULT` 关键字，以符合 Tailwind CSS V4 的约定或要求。
</Callout>
